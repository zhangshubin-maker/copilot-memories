- 代码要有组织性；文件变大时再拆分。
- 拆分优先用 mixin，但只有出现多个清晰 mixin 切片或真实复用边界时才拆。
- 不要把主组件抽空成只挂一个 page mixin 的壳；入口组件要保留主要流程。
- mixin 不合适时再考虑 js 工具类。
- 变量尽量少；命名先保证有意义，再在不损失语义的前提下简化。不要为了短而缩写成看不出含义的名字。
- 不要过度使用 normalize 一类统一边界判断的包装。
- 非必要不默认跑整仓构建校验；优先最窄验证，只有改动影响面较大或用户明确要求时再跑 build。
- 不要默认跑 H5 build 做校验；只有我明确要求，或确实没有更便宜的验证手段时才跑。

- 不要默认跑 lint 做校验；仓库里预存了大量无关 lint 报错，跑 lint 基本无用。只有我明确要求或改动文件恰好无预存报错时才跑。
